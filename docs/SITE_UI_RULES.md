# サイトUIルール

## 全体方針
- サイト全体は中学生から20代後半向けの推し活ファッション編集メディアとして、明るく生き生きしたトーンに寄せる。過度に大きな見出し、AI感のあるフォントスケール、装飾過多なラベルは避ける。
- 主要フォントはゴシック系を基本とする。特に記事内では明朝体を使わない。
- フォントは既に採用している `BIZ UDPGothic` / `Zen Kaku Gothic New` 系を維持する。UI刷新時もフォント変更で雰囲気を作らず、余白、写真、線、ラベル、カードの形で調整する。
- UI部品は丸いピル、強いシャドウ、ふわっとした白カードを多用しない。若年向けの軽さは、写真主役のカード、小さな角丸ラベル、横スクロール導線、短い見出しで作る。
- カード、タグ、検索ボックス、ボタンの角丸は原則4px前後に揃える。大きな丸みや過度な影はAI生成UIっぽく見えるため避ける。
- 参考UIは `Seventeen`、`Ray`、`non-no`、`ViVi` のように、写真、短いカテゴリラベル、ランキング、横スクロール棚を中心に組み立てる。特にトップページは `Seventeen` 寄りに、白地をベースに明るいピンク、ミント、イエロー、ラベンダーを局所的に散らす。
- 色使いはくすみローズやベージュに寄せすぎない。主アクセントは明るいピンク、補助アクセントはミントとイエローにし、画面全体が暗く大人っぽく沈まないようにする。
- Headerのブランド表示にはバッグなどのマークを置かず、文字ロゴ中心にする。
- スマホ幅のHeaderでは、サイト名が検索ボタンの影響で左右にずれないよう、左側スペースと右側検索ボタンの幅を揃えて中央列に固定する。
- スマホ幅のHeaderでは、サイト名 `oshikatsu coord` とサブコピーが画面中央に見えることを優先する。左側のカテゴリ導線を非表示にする場合も、左カラム自体はスペーサーとして残す。
- スマホ幅でブランド文字が中央列に収まらない場合は、文字サイズを無理に下げず、ブランド表示の内側だけを横スクロール可能にする。ページ全体の横スクロールは発生させない。
- Headerの最上部ブランド行は、PC幅では `CATEGORY` を左、サイト名を中央、検索ボタンを右に置く。前回のようにロゴ、CATEGORY、検索を中央付近に寄せる配置へ戻さない。
- Headerとは、文脈により `HOME`、`LOOK`、`COLOR` などが並ぶナビゲーション行を指す場合がある。ブランドロゴ行とは分けて扱う。
- スマホ幅のHeaderナビゲーション行は、項目を無理に縮めず、横スクロールで操作できるようにする。`HOME` から `FEATURE` までを1行で並べ、ページ全体の横スクロールは発生させない。
- PC幅のHeaderナビゲーション行は、中央に寄せすぎない。画面端まで広げすぎず、中央付近の適度な幅の中で項目間隔を広めに取る。右端に寄せるのではなく、ナビ全体の幅を広げて均等に散らす。
- ナビとFooterには存在しないカテゴリへのリンクを残さない。
- 「このサイトについて」ページと、そのHeader/Footer導線は置かない。
- Header右側の虫眼鏡ボタンは記事一覧へのリンクにしない。押すと検索ボックスが上から開く。
- 検索ボックスでは入力中に結果を直下表示しない。キーワードを入力して検索ボタンを押したときだけ、検索結果ページへ遷移する。
- 検索結果ページは `/search?q=キーワード` 形式にし、タイトルまたは本文など検索対象テキストにキーワードを含む記事を一覧表示する。
- 検索は記事タイトルだけでなく、記事本文、要約、見出し、色、シーン、アイテム情報を含む検索用テキストを対象にする。
- Footerのリンクは必要最小限にし、原則として「記事一覧」「カテゴリ一覧」だけを置く。ルール・マナー、お買い物ガイドへの個別導線はFooterには置かない。
- Footerは下部に大きな余白を残さず、リンク数が少ない状態に合わせてコンパクトな上下余白にする。特に「記事一覧」「カテゴリ一覧」の下に不要な余白を残さない。
- UIに変化を加える場合は、PC表示だけで判断せず、必ずスマホ幅のレイアウトも確認する。ヘッダー、横スクロール棚、商品カード、記事本文の余白と折り返しを重点的に見る。
- スマホの記事ページでは、記事画像が本文カラムの横幅を押し広げないようにする。記事本文カラムには `min-w-0` を置き、画像・figureには `max-width: 100%` 相当の指定を入れる。

## ターゲット
- 主な読者は中学生から20代後半までの女性とする。
- 推し活やバウンドコーデを、コスプレや公式衣装再現ではなく、日常服として自然に楽しみたい読者を中心にする。
- 「推しをそのまま着る」のではなく、色、素材、小物、シルエットで推しの空気を普段着に混ぜたい読者を想定する。
- ライブ、イベント、コラボカフェ、推し会、本人不在の誕生日会、普段のお出かけで使える服装を探している読者に向ける。
- 推し色コーデやバウンドコーデに興味はあるが、やりすぎ感、子どもっぽさ、公式再現感は避けたい読者を想定する。
- プチプラやECも使いながら、安っぽく見えない組み方、日常にも戻せる服選び、現場で困らない実用性を重視する。
- 著作権、公式素材、非公式表現の距離感に配慮し、安心して参考にできる推し活ファッションメディアとして設計する。
- 文章は初心者にもわかりやすく、ただし幼くなりすぎない編集メディア調にする。本文は敬体を基本にしつつ、語尾が単調にならないよう短文、体言止め、軽い常体を混ぜる。

## 参考媒体と編集トーン
- ターゲット変更後の主参考は、`Seventeen`、`Ray`、`non-no` とする。中高生への近さ、甘めファッションの見せ方、大学生以上にも読める整理力を組み合わせる。
- 補助参考は、`ViVi`、`CanCam`、`MERY`、`Oshicoco`、`WEGO推し活`、`OshiKra` とする。ViViは勢い、CanCamはHowTo整理、MERYは若年向け軽さ、Oshicoco/WEGO/OshiKraは推し活語彙と商品導線の参考にする。
- `Oggi`、`CLASSY.`、`Sheage` は20代後半以上に寄りやすいため、主参考にはしない。余白、整理、安心感の補助としてだけ扱う。
- 全体の編集トーンは、Seventeenの近さ、Rayの甘さ、non-noの整理力を軸にする。推し活記事ではOshicoco/WEGOの現場感を少し足す。
- UIのデザイン読みは「中学生から20代後半の推し活ファッション読者向け編集メディア」とする。過度な高級誌風、無機質な余白過多、ニュースサイト風の文字密度は避ける。

## 文章ルール
- 本文は敬体を基本にする。ただし全文を「です・ます」で揃えず、短文、体言止め、軽い常体を混ぜてリズムを作る。
- 語尾は「おすすめです」「できます」「なります」を連続させない。「入れてみて」「合わせると自然」「まずは小物から」など、軽い友だち感のある言い方を混ぜる。
- 中高生にも届くよう、1段落は短めにする。1段落は原則2-3文まで、1文は長くしすぎない。
- 見出しは「悩み/シーン + 叶うこと」で作る。例: 「ライブの日に浮かない推しカラー小物」「学校帰りでもできる、さりげなバウンドコーデ」。
- LOOK記事の導入は、現場前の気持ち、写真を撮る場面、推し色を入れたい理由から始める。説明の前に読者の気分へ接続する。
- 推し活語彙は自然に使う。使用可: `推し色`、`メンカラ`、`現場`、`参戦`、`遠征`、`ぬい`、`アクスタ`、`トレカ`、`痛バ`、`概念`、`オタ友`、`双子コーデ`。
- 熱量表現は、読者の気持ちに寄り添う範囲で使う。例: `推しに会える日`、`大切な日`、`お守りみたいな一着`、`推しを近くに感じる`。
- 強すぎる断定や商業臭い表現は避ける。不可: `絶対買うべき`、`ファンサ確定`、`推しに見つかる`、`今すぐポチって`、`モテ確定`。
- 幼くなりすぎる表現は避ける。不可: 絵文字連打、`きゅんきゅん♡`、`姫すぎる`、`優勝すぎ` の乱用、語尾の `だよぉ`。
- AIっぽい定型文を禁止する。不可: `いかがでしたか？`、`ぜひ参考にしてみてください` だけの締め、`本記事では〜について解説します`、`〜することができます` の多用、`魅力を最大限に引き出す`、`ワンランク上の`。
- 権利面の距離感を保つため、`公式衣装風`、`完全再現`、`公式っぽい` などの表現は使わない。色、素材、空気感、ステージのきらめき、音のイメージなどへ言い換える。
- ファンサに関する表現は断定しない。`もらえる` ではなく、`写真に残したくなる`、`現場までの気分が上がる`、`自分の気持ちが整う` に寄せる。

## トップページ
- トップページ冒頭には、横長の特集記事カルーセルを配置する。
- カルーセルは4件の特集記事を自動で切り替え、ユーザーが前後ボタンとインジケーターで手動切り替えできるようにする。
- カルーセルは写真だけを見せる。タイトル、説明文、色チップなどのテキスト情報はカルーセル内に置かない。
- スマホ幅ではカルーセルを縦長寄りにし、記事画像の中央カラーパネルやコーデ写真が横長バナー枠で大きく切れないようにする。タブレット以上では横長バナーへ戻す。
- トップページには、サイトコンセプトの長い説明文を置かない。
- トップページのファーストビューは、注目記事や特集画像を主役にし、若年読者がすぐに押せる導線を置く。説明文より、画像、カテゴリ、ランキングへの入口を優先する。
- カルーセル下には、カテゴリごとのトピックコーナーを順番に掲載する。
- 各カテゴリコーナーでは、そのカテゴリの記事を最大3件並べ、サムネイルは正方形で表示する。
- トップページのカテゴリごとのトピックコーナーでは、各タイトル上の `New articles` などの英字ラベルを表示しない。
- トップページのカテゴリタイトルと「よく読まれている記事」見出しは、既存のゴシック系フォントを使い、斜体や明朝体にはしない。太めのポップな文字、ピンク文字、黄色マーカー、ミント系の影などで若さを出す。
- 「よく読まれている記事」のような日本語の棚見出しは、英字の `LOOK` / `COLOR` より視覚的に大きく見えやすいため、専用に少し小さめのサイズを指定して他の棚タイトルと揃える。
- スマホ幅では、各カテゴリコーナーの3件は縦積みにせず、横スクロールで表示する。
- スマホ幅のトピック横スクロールは、左端が詰まって見えないように本文余白より少し広めの開始余白を取る。
- スマホ幅のカルーセルや記事棚は、自動送りよりもスワイプしやすさ、現在位置のわかりやすさ、カード間余白を優先する。
- トップページに「今気になる言葉から探す」ブロックは置かない。タグ検索やキーワード検索はHeader検索または検索結果ページへ集約する。
- カテゴリで絞り込んだページ上部のジャンル名（例：LOOK、COLOR）は、ホーム棚見出しとは別に、コンパクトなゴシック系タイトルとして表示する。
- 記事数が3件未満のカテゴリでは、不足分にグレーの正方形プレースホルダーを表示する。
- 「シーン別コーデ」はカテゴリとして扱わない。ライブ、イベント、コラボカフェなどの場面情報は記事内のタグ・文脈として扱い、カテゴリ棚やナビには出さない。
- 「初心者向けノウハウ」「ルール・マナー・著作権」のような旧粒度のカテゴリは復活させず、内容に応じて `FEATURE` または `GUIDE` に分類する。
- 「よく読まれている記事」はトップページ下部に残す。
- 「目的から探す」ブロックはトップページには置かない。

## カテゴリ運用
- コンテンツ分類は `LOOK`、`COLOR`、`STYLE`、`SCENE`、`GUIDE`、`FEATURE` の6つにする。
- CSVでは `articleType` を主分類、`featureType` を細分類、`menuLabel` を画面表示ラベルとして扱う。基本的に `menuLabel` は6分類名と一致させる。
- 旧カテゴリの「推し別コーデ」は `LOOK`、「推し色コーデ」は `COLOR`、「テイスト別コーデ」は `STYLE`、「シーン系記事」は `SCENE`、「買い物・マナー・アイテム選び」は `GUIDE`、「読み物・基礎特集」は `FEATURE` へ再分類する。
- カテゴリ一覧やトップページのトピック棚には、旧カテゴリ名を表示しない。
- 記事本数の初期目標は合計100本とする。内訳は `LOOK: 42`、`COLOR: 16`、`STYLE: 10`、`SCENE: 12`、`GUIDE: 6`、`FEATURE: 14`。

## 記事ページ
- 新CSVスキーマでは `publishedAt` / `updatedAt` を持たないため、記事ページ上部に公開日・更新日は表示しない。
- いいね数はサイト内に表示しない。
- 記事内のタイトル、見出し、本文、関連記事カードには明朝体を使わず、ゴシック系フォントで統一する。
- 記事タイトルは参考メディアサイトに近い控えめな編集見出しサイズにする。記事カードは通常約16-17px、先頭カードでも約20-23px、記事詳細H1はPCでも約36px前後を上限にする。
- 記事タイトルは、感情語、用途、シーン、推し活語彙を組み合わせる。例: `推し色をさりげなくまとえる、春の現場LOOK`、`学校帰りでも浮かない水色バウンドコーデ`。
- 記事上部には、カテゴリ、タグ、短いリードを置き、読者が「自分向けの記事か」をすぐ判断できるようにする。
- 長い記事では、冒頭に「この記事でわかること」相当の短いまとめを置く。ただし装飾や説明文を増やしすぎない。
- 記事サムネイル上に `ARTICLE 01` などの番号ラベルを表示しない。
- 記事No.はサイト内に表示しない。カード、ランキング、記事詳細、サムネイル上に `No. 1` などを出さない。
- `LOOK` と `COLOR` は従来のカラー・コーディネート提案型構成を適用する。
- `LOOK` 記事は、導入、色と雰囲気、コーディネートイメージ、アイテム構成、推し活ポイント、関連記事の順を基本にする。
- `LOOK` 記事の本文では、色、シルエット、小物、推し活アイテムの持ち歩きやすさ、写真映え、現場での動きやすさを扱う。
- `STYLE`、`SCENE`、`GUIDE`、`FEATURE` は、LOOK/COLORとは記事構成と写真方針を大きく変える。本文は読み物として成立する構成にし、カラー表、商品カード、コーデ画像、アイテム構成を機械的に出さない。
- `SCENE` 記事は、ライブ、遠征、コラボカフェ、学校帰り、休日など、読者の1日の動きを起点に構成する。
- `GUIDE` 記事は、検索ワード、サイズ選び、収納、マナーなど、実用手順を短いブロックで見せる。
- `FEATURE` 記事は、サイト思想や推し活の楽しみ方を扱うが、説教調にしない。読者の気持ちが軽くなる読み物にする。
- `GUIDE` と `FEATURE`、およびメイン・サブ・アクセントがすべて「複数」の記事本文には、カラーパレットコンポーネントを表示しない。
- カラーパレットは配色チャートとして表示し、目安比率はメイン60%、サブ25%、アクセント10%、調整5%にする。調整カラーがない記事では、残り5%を表示中の色へ按分する。
- 記事本文の `Color Palette` コンポーネントは、角丸なしの低い横長バーだけで見せる。％表記、グラフ下の色名・役割・説明文、`usage` の説明文は表示しない。
- 記事上部の「この記事のまとめ」にはハートなどの装飾アイコンを置かない。アイコン削除後の余白も残さない。
- 記事本文のコーデ画像セクション見出しは「コーディネートイメージ」にする。画像上や右下に「コーデのポイント」などの説明オーバーレイを重ねない。
- コーデ画像の生成表現に関する注釈文は、記事本文中のセクション注釈として表示しない。
- おすすめアイテムには `OSHIKATSU SELECT` などの仮EC名ラベルや、仮データである旨の読者向け注釈を表示しない。
- スマホ幅のおすすめアイテムは、カードが潰れないよう横スクロールで表示する。カード幅は200px前後を下限にし、画像枠、文字サイズ、ボタン幅を固定気味にして、商品名・価格・ボタンが重ならないようにする。
- 「アイテム構成」セクションは、構成イラスト画像と手書き風の短い解説メモを組み合わせて表示する。
- 記事本文では「シーン別の着方」セクションを表示しない。
- `ngPoints` は制作・チェック用の内部情報として扱い、読者向け記事本文には表示しない。
- 記事本文の最後には関連記事を3件並べて表示する。
- スマホ幅では記事画像上に説明ボックスを重ねず、画像下へ流して本文や写真を隠さない。
- スマホ幅の記事ページでは、タグ、関連記事、商品カードを横スクロールで扱えるようにし、本文を読む流れを止めない。
- 商品導線は広告っぽく浮かせない。記事中では「このコーデに近づけるアイテム」や「似た雰囲気を探す」程度の自然な導線にする。

## CSVと表示データ
- 記事管理は `INITIAL_ARTICLES.csv` を正とし、記事追加・更新時はCSVへ書き込む。
- CSV列は `No`, `articleType`, `featureType`, `menuLabel`, `title`, `slug`, `characterName`, `workName`, `mainColor`, `subColor`, `accentColor`, `moodKeywords`, `scene`, `itemTypes`, `tags`, `affiliatePriority`, `status`, `memo` を基本とする。
- いいね数など、実データとして管理しない指標は表示しない。
- 記事サムネイルは記事内で使う画像を流用する。
